# CEX Trading Engine — Learning Path

A curriculum for building a production-grade centralized exchange matching engine, structured around the Exchanges case study (single-pair, in-memory, HTTP API) but covering enough theory to defend the design in an interview and extend it later.

## How to read this

Each file is tagged at the top:

- **[CHALLENGE]** — required for the case study (5–8 hour budget). Read and implement.
- **[NEXT STEP]** — production concerns explicitly out of scope for the challenge. Read for context, mention in the README's "what I'd do next with more time" section, do **not** implement.

Every topic has theory, trade-offs, and runnable Go samples in `samples/`.

## Suggested study order

1. **Skim 01 & 02** for vocabulary (you'll see "BBO", "maker/taker", "level" everywhere).
2. **Read 05 first** before writing any matching code — choosing the wrong numeric type breaks everything downstream and is painful to retrofit.
3. **Read 03 & 04 carefully.** These are the correctness-critical pieces. Write tests as you read.
4. **Read 06** once you have a working sequential engine; concurrency is the last layer to add.
5. **07, 08** are short polish topics — pick them up before the README write-up.
6. **09, 10** are interview ammunition. Read them after you've shipped the code so you can speak to "what next" with confidence.

## Reference engines

- **Binance** — proprietary, fee-tiered, very high throughput. Public docs describe the API surface, not the engine internals.
- **Hyperliquid** — perpetuals DEX with fully on-chain order book. Engine is Rust, single-threaded per market, deterministic by construction (consensus replays the same order sequence on every node).
- **CME Globex** — the canonical price-time priority reference; their self-match prevention spec (SMP) is the industry model.
- **LMAX Disruptor** — open-source ring-buffer architecture; the canonical "single-writer" exchange design that influenced most modern engines.

## Sample code layout

```
samples/
├── decimal/    — fixed-point vs shopspring/decimal demos
├── orderbook/  — price-level tree + FIFO queue
├── matching/   — full matching loop with partial fills
├── stopbook/   — armed stop orders + trigger scan
└── engine/     — minimal end-to-end engine wiring all of the above
```

Each sample is a self-contained `main` package — `go run samples/<name>/<file>.go` to play with it.

## Five things that score points in the interview

1. **Decimal-correct from line 1.** Float bugs are the #1 reason candidates fail correctness review.
2. **Engine is import-free of HTTP/JSON.** Pure function of state + commands. Drive it from tests.
3. **Determinism is engineered, not hoped for.** No goroutine-fanout, no map iteration where order matters, IDs and timestamps generated by the engine.
4. **One writer.** Boring `sync.Mutex` over the engine. Don't reach for lock-free unless you can defend it.
5. **Trade-offs articulated.** "I picked X over Y because Z" beats "I used X" every time.
